# ADR-003: Separación de la capa de Mappers como componente independiente

**Fecha:** 2026-04-14  
**Estado:** Terminado

---

## Contexto

El sistema Piedra Azul maneja entidades JPA que contienen información
sensible y campos de persistencia que no deben exponerse directamente
al cliente. Por ejemplo, la entidad `Paciente` contiene campos internos
de la base de datos (`id`, relaciones JPA con `@OneToMany`) y la entidad
`Cita` referencia objetos completos de `Medico` y `Paciente` que, si se
serializaran directamente, expondrían datos clínicos de terceros en cada
respuesta.

La solución inicial fue usar DTOs (`PacienteRequest`, `PacienteResponse`,
`CitaRequest`, `CitaResponse`) para controlar qué datos entran y salen
del sistema. Sin embargo, surgió una segunda decisión: ¿dónde vive la
lógica de conversión entre una entidad y su DTO?

Las opciones evaluadas fueron:

- **Opción A:** Hacer la conversión dentro del propio servicio
  (`CitaService`, `MedicoService`), mezclando lógica de negocio con
  lógica de transformación de datos.
- **Opción B:** Hacer la conversión dentro del controlador, antes de
  retornar la respuesta HTTP.
- **Opción C:** Crear una clase `Mapper` dedicada por módulo
  (`CitaMapper`, `MedicoMapper`, `PacienteMapper`) que encapsula
  exclusivamente la lógica de transformación.

Las opciones A y B violaban el Principio de Responsabilidad Única: un
servicio que convierte DTOs también está haciendo trabajo que no le
corresponde, y cualquier cambio en la estructura del DTO obligaba a
modificar clases de negocio que no deberían verse afectadas.

---

## Decisión

Se adoptó la **Opción C**: cada módulo del backend cuenta con su propia
clase `Mapper` que centraliza toda la lógica de conversión entre entidades
y DTOs, siguiendo el Principio de Responsabilidad Única.

La estructura resultante en cada módulo es:
cita/
├── CitaController.java     → recibe Request, retorna Response
├── CitaService.java        → orquesta lógica de negocio con entidades
├── CitaMapper.java         → convierte entre Cita ↔ CitaRequest/CitaResponse
├── CitaRepository.java     → acceso a datos
├── dto/
│   ├── CitaRequest.java
│   └── CitaResponse.java
└── entity/
└── Cita.java

El flujo de datos sigue este camino en cada operación:

- **Entrada:** `CitaRequest` (JSON del cliente) → `CitaMapper.toEntity()`
  → `Cita` (entidad JPA) → `CitaRepository.save()`
- **Salida:** `Cita` (entidad JPA) → `CitaMapper.toResponse()`
  → `CitaResponse` (JSON al cliente)

Los mappers se inyectan en los servicios mediante `@Component` de Spring,
siguiendo el mismo ciclo de vida Singleton que el resto de componentes
del contenedor de inversión de control.

---

## Consecuencias

### Positivas

- **Responsabilidad única cumplida:** `CitaService` solo orquesta reglas
  de negocio (validación de cruces de horario, verificación de
  disponibilidad del médico). No sabe nada de cómo se serializa una cita
  para el cliente.
- **Cambios de contrato aislados:** si el frontend solicita un nuevo campo
  en `CitaResponse`, el cambio se hace únicamente en `CitaMapper` y en el
  DTO, sin tocar la lógica de negocio ni la entidad de base de datos.
- **Seguridad por diseño:** campos sensibles de las entidades (relaciones
  JPA, campos internos) nunca llegan al cliente porque el mapper
  selecciona explícitamente qué campos incluir en el DTO de respuesta.
- **Testabilidad:** los mappers son clases simples sin dependencias de
  infraestructura, lo que permite escribir pruebas unitarias de
  transformación sin necesidad de levantar el contexto de Spring ni
  conectarse a la base de datos.

### Negativas / Riesgos

- **Proliferación de clases:** cada módulo agrega al menos una clase
  Mapper adicional. En un proyecto con muchos módulos esto incrementa
  el número total de archivos a mantener, lo que puede dificultar la
  navegación del código para desarrolladores nuevos.
- **Sincronización manual:** los mappers deben actualizarse manualmente
  cada vez que cambia una entidad o un DTO. No existe un mecanismo
  automático que detecte que `CitaMapper` quedó desactualizado si se
  agrega un campo nuevo a la entidad `Cita`, lo que puede generar bugs
  silenciosos donde el campo nuevo simplemente no aparece en la respuesta.
- **Duplicación potencial:** si dos módulos necesitan representar la misma
  entidad de forma diferente (por ejemplo, `Medico` resumido en una
  respuesta de cita vs. `Medico` completo en su propio endpoint), se
  pueden acumular múltiples versiones de la misma transformación en
  distintos mappers sin una estrategia clara de reutilización.
